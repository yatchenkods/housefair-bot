# Техническое задание: Telegram‑бот для управления домашними задачами

## 1. Введение

Цель — разработать Telegram‑бота для учёта и честного распределения домашних задач между членами семьи (ориентировочно 2–6 человек), с базовыми возможностями просмотра, назначения, выполнения задач и напоминаний. Бот должен быть реализован на Python, работать в Docker‑контейнере и использовать БД с возможностью дальнейшей миграции с SQLite на PostgreSQL.

Важное архитектурное требование: **Telegram‑бот не взаимодействует с базой данных напрямую**. Все операции с данными выполняются через отдельный API‑слой (backend‑сервис), который предоставляет HTTP/REST интерфейс.

Данный документ описывает функционал модуля управления задачами (без кулинарного ассистента и отчётных/игровых модулей).

***

## 2. Архитектура и стек

- Язык: **Python 3.11+**  
- Telegram‑бот: **python‑telegram‑bot 21.x**  
- API‑слой (backend‑сервис): **FastAPI**  
- База данных:
  - MVP: **SQLite** (файл в Docker volume, подключён только к backend‑сервису)
  - В перспективе: **PostgreSQL**, без изменения бизнес‑логики  
- ORM‑слой (используется только в backend‑сервисе): **SQLModel** или **SQLAlchemy ORM**  
- Контейнеризация: **Docker**, возможное использование docker‑compose / Kubernetes

### 2.1 Разделение на сервисы

1. **Сервис Telegram‑бота**  
   - Отвечает за взаимодействие с Telegram API.  
   - Обрабатывает команды пользователей, формирует запросы к API‑слою и отображает ответы.  
   - Не содержит кода доступа к БД и не подключается к БД.

2. **API‑backend (сервис доменной логики)**  
   - Реализован на FastAPI.  
   - Содержит модели данных, бизнес‑логику и ORM‑слой.  
   - Предоставляет REST‑эндпоинты для операций с семьями, участниками и задачами.  
   - Единственный компонент, имеющий доступ к БД.

Взаимодействие: бот вызывает методы API‑backend по HTTP (например, `http://backend:8000/api/chores/...`), передавая необходимые данные (ID пользователя, ID семьи, параметры задачи и т.д.), а затем на основе ответа формирует сообщения в Telegram.

***

## 3. Работа с базой данных

### 3.1 Общие требования

1. **Прямой доступ к БД разрешён только API‑backend‑сервису.**  
   - Сервис бота не знает о структуре БД и не содержит строк подключения к базе.  
   - Любые операции создания/чтения/обновления/удаления сущностей выполняются через HTTP‑запросы к API‑слою.

2. В API‑backend используется ORM‑слой (SQLModel/SQLAlchemy) для работы с SQLite и PostgreSQL через единый код.  

3. Строка подключения к БД задаётся переменной окружения `DATABASE_URL` в контексте backend‑сервиса, например:
   - `sqlite:///./data/database.db`
   - `postgresql+psycopg://user:password@host:5432/dbname`

4. Структура БД создаётся и изменяется на основе ORM‑моделей и миграций.

### 3.2 Отказоустойчивость и хранение данных

**Для MVP (SQLite, только в backend‑сервисе):**

- Файл базы хранится в персистентном Docker volume (`./data:/app/data`) у backend‑контейнера.  
- Включён WAL‑режим.  
- Настроено ежедневное резервное копирование файла БД в `/app/data/backups` с ограничением по количеству бэкапов (например, 30).

**При миграции на PostgreSQL:**

- Меняется `DATABASE_URL` и конфигурация backend‑сервиса.  
- Telegram‑бот продолжает работать с теми же HTTP‑эндпоинтами; его логика не меняется.  

***

## 4. Модель данных (API‑слой, только задачи)

Модели `families`, `members`, `chores` используются в backend‑сервисе и доступны боту только через API (DTO/JSON‑ответы).

### 4.1 `families`

- `id` — PK  
- `name` — название семьи/группы  
- `chat_id` — Telegram chat ID  
- `created_at` — дата создания  
- `timezone` — часовой пояс (например, `Europe/Moscow`)  
- `settings` — JSON‑настройки (время напоминаний, режим распределения и т.п.)

### 4.2 `members`

- `id` — PK  
- `family_id` — FK → `families.id`  
- `user_id` — Telegram user ID  
- `username` — Telegram username  
- `display_name` — отображаемое имя  
- `role` — `admin` / `member`  
- `joined_at` — дата присоединения  
- `preferences` — JSON (опционально, на будущее)

### 4.3 `chores`

- `id` — PK  
- `family_id` — FK → `families.id`  
- `title` — название задачи  
- `description` — описание (опционально)  
- `chore_type` — `one_time` / `daily` / `weekly` / `monthly`  
- `category` — категория (посудa, уборка и др.)  
- `assigned_to` — FK → `members.id` (или `NULL`)  
- `created_by` — FK → `members.id`  
- `created_at` — дата создания  
- `due_date` — срок выполнения (опционально)  
- `completed_at` — дата выполнения (опционально)  
- `status` — `pending` / `completed` / `overdue`  
- `photo_url` — идентификатор/ссылка на фото

## 5. Функциональные требования: управление задачами

Во всех пунктах ниже подразумевается, что бот вызывает соответствующий HTTP‑эндпоинт API‑backend‑а, передаёт необходимые параметры и отображает пользователю результат.

### 5.1 Создание задач

- Команды бота: `/addchore`, `/task`.  
- Бот собирает у пользователя данные (название, описание, тип, категория, фото) и отправляет POST‑запрос в API:

  Пример: `POST /api/chores`  
  Тело: JSON `{ family_id, created_by, title, description, chore_type, category, photo_url }`.

- API‑backend создаёт запись в БД и возвращает данные созданной задачи.  
- Бот отображает пользователю подтверждение и краткую информацию по задаче.

### 5.2 Назначение задач

- Бот предлагает режим назначения (ручной, случайный, ротация, свободная задача).  
- После выбора режима бот отправляет запрос в API:

  - либо обновление существующей задачи  
    `PATCH /api/chores/{id}` с `assigned_to`, `due_date`  
  - либо отдельный метод распределения, если логика сложнее  
    `POST /api/chores/{id}/assign`  

- API‑backend выполняет бизнес‑логику (выбор участника, проверка прав и т.п.) и возвращает обновлённую задачу.  
- Бот уведомляет назначенного участника и семейный чат.

### 5.3 Просмотр задач

- `/mytasks`, `/alltasks`, `/pending`, `/history`.  
- Бот делает GET‑запросы к API, например:
  - `GET /api/chores?family_id=...&assigned_to=me&status=pending`  
  - `GET /api/chores?family_id=...&status=pending`  
  - `GET /api/chores/history?family_id=...&from=...&to=...`  

- API‑backend возвращает список задач в JSON, бот форматирует и выводит их в сообщения/inline‑списки.

### 5.4 Отметка выполнения

- `/done <id>` или нажатие inline‑кнопки «Выполнено».  
- Бот вызывает, например:  
  `POST /api/chores/{id}/complete` с информацией о пользователе, при необходимости — с фото (сначала загрузка/привязка `photo_url`).  
- API‑backend:
  - проверяет права исполнителя;  
  - обновляет `status = completed`, `completed_at = now()`, `photo_url` (если новое фото);  
  - возвращает обновлённую задачу.  
- Бот отправляет пользователю и/или в чат подтверждение.

### 5.5 Напоминания

- Логика расписаний может быть реализована:
  - либо в боте (scheduler), который раз в N минут дёргает API‑backend за списком просроченных/актуальных задач;
  - либо в самом API‑backend (планировщик), который уведомляет бота/чат.  

Ключевое требование остаётся: **даже при отправке напоминаний список задач и их состояния всегда берётся из API‑backend, а не из прямого доступа к БД со стороны бота.**

***

## 6. Нефункциональные требования

- Telegram‑бот и API‑backend — отдельные процессы/контейнеры.  
- Бот не содержит строки подключения к БД и к ORM‑моделям.  
- Время ответа API на типовые операции (создание/чтение/обновление задач) — до 200–300 мс при нормальной нагрузке; итоговый отклик бота — до 2 секунд.  
- Переключение SQLite → PostgreSQL требует изменений только в конфигурации backend‑сервиса и миграций; Telegram‑бот продолжает работать с теми же HTTP‑эндпоинтами.
